Bài 4.4: Multi-agent orchestration

⏱ ~8 phút đọc Level 4 Advanced
💡
Nguyên lý

Một agent làm tuần tự từng bước. Nhiều agents làm song song các bước độc lập. Kết quả: task phức tạp hoàn thành nhanh hơn 3-5x, với chất lượng cao hơn vì mỗi agent chuyên sâu một vai trò. Đây là cách Boris Cherny tại Anthropic tạo 100 PRs/tuần.

Tại sao multi-agent?

Hãy tưởng tượng bạn cần implement một feature mới: research giải pháp → viết code → viết tests → review code → update docs. Một agent làm tuần tự: 45 phút. Bốn agents song song: 15 phút, mỗi agent làm việc mình giỏi nhất.

Nhưng performance không phải lý do duy nhất. Multi-agent còn giải quyết vấn đề context window. Khi task phức tạp, một agent dễ bị "lạc" trong context quá dài. Subagents làm nhiệm vụ hẹp hơn, ít noise hơn, ra kết quả tốt hơn.

Agent types trong Claude Code

planner researcher fullstack-developer tester code-reviewer debugger docs-manager general-purpose

Mỗi agent type có "vai diễn" khác nhau — được định nghĩa qua system prompt hoặc skill. planner phân tích và chia tasks. researcher chỉ tìm kiếm, đọc, không viết code. tester chỉ sở hữu test files. code-reviewer chỉ đọc và comment.

Quan trọng: mỗi agent cần file ownership rõ ràng. Khi 3 developer agents cùng chạy song song, ai sở hữu file nào? Nếu không định nghĩa rõ, conflict xảy ra.

3 orchestration patterns

Sequential Pipeline

Agents chạy lần lượt, output của agent trước là input của agent sau. Dùng khi mỗi bước phụ thuộc vào kết quả bước trước.

plan → implement → test → review

Parallel Fan-out/Fan-in

Lead agent phân task → nhiều worker agents chạy đồng thời → lead agent tổng hợp kết quả. Dùng khi tasks độc lập với nhau.

lead → [dev-A, dev-B, dev-C] → merge

Hybrid

Kết hợp cả hai. Research song song → plan tuần tự → implement song song → review tuần tự. Pattern phổ biến nhất cho feature development đầy đủ.

research(×3) → plan → impl(×3) → review

Git worktrees: Zero merge conflicts cho parallel agents

Vấn đề lớn nhất khi nhiều agents code song song: merge conflicts. Giải pháp: mỗi agent làm việc trong một git worktree riêng — một copy của repo với branch riêng.

# Setup 3 worktrees cho 3 developer agents
git worktree add ../feature-auth -b feature/auth
git worktree add ../feature-payment -b feature/payment
git worktree add ../feature-dashboard -b feature/dashboard

# Mỗi agent chạy trong một worktree
# Agent 1: cd ../feature-auth && claude
# Agent 2: cd ../feature-payment && claude
# Agent 3: cd ../feature-dashboard && claude

# Sau khi xong, merge về main
git worktree remove ../feature-auth
git merge feature/auth

Với worktrees, ba agents có thể edit cùng file paths mà không conflict — vì chúng ở trong directories khác nhau. Khi merge về main, conflicts có thể xảy ra nhưng tại thời điểm bạn đã review xong, không phải giữa chừng.

Cách spawn subagent trong Claude Code

Trong session Claude Code, bạn có thể spawn subagent bằng cách mô tả task:

# Trong CLAUDE.md hoặc trong prompt:
"Spawn một researcher agent để tìm hiểu best practices
cho PostgreSQL indexing, báo cáo vào plans/research/"

# Claude sẽ:
# 1. Tạo Task tool call với prompt cho researcher
# 2. Researcher agent chạy, đọc docs, viết report
# 3. Main agent nhận kết quả, tiếp tục với đầy đủ context

Subagent delegation giảm 40-60% main context usage — researcher đọc 50 trang docs rồi tóm tắt thành 2 trang cho main agent. Main agent không cần đọc 50 trang đó.

Real workflow: 100 PRs/tuần

Boris Cherny, Software Engineer tại Anthropic, chia sẻ workflow thực tế: mở 10-15 terminal windows, mỗi window là một Claude Code session với một task. Buổi sáng setup tasks, buổi chiều review kết quả và merge PRs tốt. Không phải mọi PR đều được merge — nhưng throughput tổng thể tăng đột biến.

Key insight: parallelism ở cấp độ task, không phải token. Không cần một AI siêu mạnh — cần nhiều AI chạy song song trên nhiều tasks độc lập.

Agent Teams: Coordination có tổ chức

Với /team skill, bạn có thể tạo team agents có cấu trúc hơn:

  • Lead agent: Nhận task tổng, phân task cho workers qua TaskCreate, đợi kết quả
  • Worker agents: Claim task, implement, báo cáo qua SendMessage
  • Communication: Agents nhắn tin cho nhau qua built-in messaging system
# Lead agent tạo tasks cho team
TaskCreate(
  title="Implement user authentication",
  description="Build JWT auth. File ownership: src/auth/*",
  assigned_to="developer-1"
)
TaskCreate(
  title="Write auth tests",
  description="Test all auth flows. File ownership: tests/auth/*",
  assigned_to="tester-1"
)
# Tester waits for developer to complete first

Limits và best practices

Số lượng concurrent agents: Tối đa 5-10 agents đồng thời cho stability. Nhiều hơn không có nghĩa là nhanh hơn — API rate limits và context management trở thành bottleneck.

File ownership là bắt buộc: Mỗi agent phải có danh sách files mình được phép edit. Không bao giờ để hai agents edit cùng file — ngay cả với worktrees.

Task granularity: Task quá nhỏ (5 phút) = overhead quản lý không xứng. Task quá lớn (4 giờ) = agent dễ lạc hướng. Target 30-90 phút per task.

⚠️
Khi nào không dùng multi-agent

Nếu task có dependencies phức tạp giữa tất cả các phần, multi-agent tạo thêm overhead hơn là tiết kiệm thời gian. Tasks cần nhiều back-and-forth với bạn (creative work, design decisions) cũng phù hợp hơn với single agent. Multi-agent tốt nhất cho tasks có thể decompose thành independent subtasks rõ ràng.

🔧
Thực hành

Thử sequential pattern với 4 bước: (1) Mở Claude Code, nói "Spawn một researcher agent, tìm hiểu 3 best practices để tối ưu React component performance, viết report ngắn." (2) Đọc report. (3) Nói "Dựa trên report vừa rồi, implement một React component demo ứng dụng practice đầu tiên." (4) Nói "Review code vừa viết, check performance issues." Quan sát cách context được truyền giữa các steps mà không cần bạn copy-paste.
